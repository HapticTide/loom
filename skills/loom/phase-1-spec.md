# Phase 1: 需求协商

## 目标

将用户的模糊意图**扩展**为结构化的 `spec.md`。是扩展，不是形式化——Spec 应比用户最初描述的更丰富、更有野心。

> "Without the planner, the generator under-scoped: given the raw prompt, it would start building without first speccing its work, and end up creating a less feature-rich application." — Anthropic

## 输入 / 输出

- **输入**: 用户的对话、项目上下文（现有代码、技术栈）
- **输出**: `.loom/<task>/spec.md`

## 协作模式

**交互式** — 与用户多轮对话。不要自行假设需求，通过提问厘清。

### 交互节奏

- **每轮只问一个问题** — 不要一口气抛出多个问题。如果一个话题需要深入，拆成多轮。
- **优先用选择题** — 给出 2-4 个选项比开放式提问更容易回答。开放式问题也可以，但选择题优先。
- **先听再扩展** — 等用户回答后再展开下一个方面，不要预设答案。

### 可视化伴侣

当你预判接下来的问题会涉及视觉内容（UI mockup、架构图、布局对比），**单独用一条消息**征求同意：

> "接下来有些内容用浏览器展示会比文字更直观（mockup、架构图等）。需要我在浏览器中辅助展示吗？"

**这条消息必须独立发送，不要和其他问题混在一起。** 等用户回复后再继续。如果用户拒绝，全程用文字。

即使用户同意，也要**按需使用**——每个问题独立判断：
- **用浏览器**: mockup、线框图、布局对比、架构图、数据流图
- **用文字**: 需求提问、概念选择、取舍列表、范围决策

判断标准：**「看」比「读」更容易理解吗？** 是就用浏览器，否则用文字。

## 核心原则

### 1. 扩展用户意图

用户给出最小意图，你主动扩展——发现用户没提到但会显著提升价值的功能。**但必须与用户确认**，不要默默加入。

### 2. 约束交付物，不约束路径

> "如果 planner 预先指定粒度化的技术细节但出了错，错误会级联到下游。更聪明的做法是约束交付物，让 agent 自行决定路径。" — Anthropic

```
✅ "用户可以在画布上拖拽放置精灵，支持网格对齐和自由放置"
❌ "使用 Canvas API 的 drawImage 方法，在 mouseup 事件中计算..."
```

### 3. 定义边界——"不做什么"比"做什么"更重要

```
❌ "支持用户管理"
✅ "支持注册、登录、个人信息编辑。不支持：角色权限、团队管理、OAuth"
```

### 4. 细节与风险成正比

高风险（隐含需求、模块间交互、架构约束）必须详细。低风险（标准 CRUD、常见 UI）可以简略。

## 协商流程

与用户讨论以下方面（按判断灵活调整，非机械执行）。记住：**每轮只问一个问题。**

### 0. 项目上下文探索（静默）

在向用户提问之前，先**自行探索**当前项目状态：
- 检查现有文件、目录结构、技术栈
- 查看 docs、README、最近的 git commits
- 识别已有的代码风格、依赖、架构模式

这一步不需要问用户，自己看完再开始提问。

### 1. 范围评估（前置门控）

**在深入细节之前，先评估范围。** 如果用户描述涉及多个独立子系统（如「构建一个平台，包含聊天、文件存储、计费和分析」），**立即标记**——不要在一个需要拆分的项目上花时间打磨细节。

如果项目过大：
- 帮用户分解为独立子项目：有哪些独立模块？它们之间什么关系？应该按什么顺序构建？
- 然后对**第一个子项目**走正常的协商流程
- 每个子项目各自走完 spec → plan → execution 的完整周期

### 2-9. 逐项讨论

2. **核心目标**: 给谁用？解决什么问题？
3. **功能模块**: 追问模糊描述，转化为用户故事（每个模块至少 3 个）
4. **范围扩展**: 识别自然延伸、体验完整性、AI 辅助可能性——列出建议让用户选择
5. **排除项**: 主动问「这个版本明确不需要什么？」
6. **隐含需求**: 根据项目类型确认用户没提但可能期望的（认证、响应式、错误处理等）
7. **项目上下文**: 现有项目时分析技术栈、代码结构、兼容性约束
8. **数据模型**: 多实体关系时描述实体和关系（不描述 schema）
9. **验收标准**: 下方详述——与用户确认「怎样算做完了？」
10. **设计语言**: 下方详述——这是整个框架中**最高杠杆的输入**

### 关键决策点：提出 2-3 个方案

在涉及**架构风格、技术选型、核心抽象**等关键决策时，不要直接推导出一个结论。而是：

1. 提出 2-3 个可行方案，附带各自的取舍
2. 给出你的推荐并说明理由
3. 让用户选择

```
✅ "数据分发有两种方案：
   A. Symlink（零拷贝，实时同步，但跨设备需要 git）
   B. Copy（独立副本，离线可用，但需要手动同步）
   我推荐 A，因为..."

❌ "数据分发用 symlink。"
```

这适用于会显著影响后续实现的决策。常规细节不需要列方案。

### 验收标准

与用户确认「怎样算做完了？」——具体、可验证的条件。

**关键原则：验收标准是交付合同，不是概要。** spec 正文中的每个行为承诺（CLI flag、API 返回值、UI 控件、配置参数、缓存策略、错误处理）都必须有对应的验收条目。如果 spec 正文写了"TTL 24 小时"，验收标准必须有一条验证 TTL 是 24 小时。如果写了"支持浅色模式"，验收标准必须有一条验证浅色模式存在。

与用户讨论：

- **核心路径**: 交付物必须满足的最基本条件是什么？
- **行为细节**: spec 正文中的每个具体参数、返回值、UI 元素、配置选项 — 逐一转化为验收条目
- **关键边界**: 异常输入、极端情况下的预期行为？
- **质量底线**: 对构建、测试、代码规范等有什么硬性要求？

**自查：** 写完验收标准后，重新通读 spec 正文，确认没有正文中描述的功能在验收标准中缺失。如果 spec 正文的某个功能确实不属于本次交付，应该移到「明确不做」section 而不是默默跳过。

根据讨论结果，在 Spec 模板的「验收标准」section 中填写具体条目。

### 设计语言

> "The prompting associated with the criteria directly shaped the character of the output." — Anthropic

设计语言定义「代码应该具有什么样的性格」。它会作为 `design-language.md` **直达**每个 Agent prompt 顶部，不经过翻译层。

与用户讨论：

- **设计宣言**: 一句话隐喻。避免空话，用参照物：
  - ✅「像 Stripe SDK——每个公开 API 都经过深思熟虑，内部实现不需要注释就能读懂意图」
  - ❌「高质量、可维护的代码」
- **架构风格**: 事件驱动 / 分层单体 / 管道式 / 挚薄控制器+富领域层
- **抽象哲学**: 组合 vs 继承？函数式管道？最小 API 表面？
- **核心约束**: 业务层不感知传输层？错误处理用 Result 还是异常？
- **代码范例参照**: 用户推崇的开源项目或风格

## Spec 模板

```markdown
# [项目/功能名称]

## 概述
[目标 + 用户画像 + 核心价值 — 2-3段]

## 功能模块

### 1. [模块名称]
[一段话描述]

用户故事：
- 作为 [角色]，我希望 [功能]，以便 [价值]

数据模型（如适用）：
- [实体]: [关键属性和关系]

### 2. [模块名称]
...

## 非功能需求
- 性能 / 安全 / 可用性（如适用）

## 技术约束
- [必须使用的技术栈/框架/兼容性要求]

## 设计语言
设计宣言：[一句话隐喻]
架构风格：[...]
抽象哲学：[...]
核心约束：[...]
代码范例参照：[如有]

## 验收标准
- [ ] [可验证的条件，描述交付物必须满足的行为或状态]
- [ ] [覆盖 spec 正文中的每个行为承诺 — CLI flag、API 返回值、UI 控件、配置参数]
- [ ] [如果 spec 正文写了具体参数（如 TTL、返回字段、UI 元素），这里必须有对应条目]

## 明确不做
- [功能排除 / 版本排除]
```

## Spec 自查

写完 spec 后，用以下清单**自查一遍**，发现问题直接修复，无需重新走流程：

1. **占位符扫描**: 有没有 "TBD"、"TODO"、未完成的 section、模糊的需求？修掉。
2. **内部一致性**: 各 section 之间有没有矛盾？架构描述和功能描述匹配吗？
3. **范围检查**: 这个 spec 的范围适合一次实现吗？还是需要拆分？
4. **歧义检查**: 有没有哪条需求可以被解读为两种意思？如果有，选一种并明确写出来。
5. **验收覆盖**: 重读 spec 正文，确认每个功能承诺在验收标准中都有对应条目。如果某个功能不属于本次交付，移到「明确不做」section。

## 用户确认

自查通过后，请用户审阅 spec：

> "Spec 已写入 `<path>` 并提交。请审阅内容，如有修改意见请告诉我，确认后进入 Phase 2。"

等待用户回复。如果用户要求修改，改完后重新跑自查。只有用户确认后才继续。

## 退出条件

- [ ] `spec.md` 已写入 `.loom/<task>/`
- [ ] Spec 自查五项全部通过
- [ ] 用户已确认内容
- [ ] 范围超出用户原始描述（有扩展）
- [ ] "明确不做"非空
- [ ] 不含实现细节（API 设计、数据库 schema、代码结构）
- [ ] 验收标准已填写（具体、可验证、覆盖核心路径）
- [ ] 设计语言已填写

完成后 → **Phase 2: 工作区生成**
